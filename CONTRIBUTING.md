# Contributing

First off, thank you for considering contributing to this project! We welcome all contributions, from bug reports and feature requests to code changes and documentation improvements.

This document outlines the standard process, guidelines, and local setup required to get you up and running quickly.


## Reporting Bugs

When submitting a bug report via our issue tracker, please include as much detail as possible to help us reproduce the issue:

* A clear and descriptive title.
* Exact steps to reproduce the behavior.
* Expected behavior versus the actual behavior.
* Environment details (OS, browser, Docker version, etc.).
* Screenshots or log output if applicable.

---

## Suggesting Enhancements

We love new ideas! If you have a feature request:

* Check existing issues to see if it has already been suggested.
* Provide a clear use case and explain how it benefits the project.
* Outline potential implementation details if you have them.

## Local Development Setup

### Prerequisites

Please ensure your local development machine has the following tools installed:

* **Docker**
* **Java** (25)
* **Maven** (3.9.x)
* **Node.js** (v25)
* **pnpm** (v11.9)
* **Angular CLI** *(Optional, but recommended)*

### Environment Initialization

#### 1. Set up Variables

Before running the application, you need to configure your environment variables. The following command copies the provided example variables into your active `.env` file.

```bash
cd docker && cp .env.example.local .env && cd -

```

#### 2. Generate Code

The application relies on code generation for both the backend and frontend APIs.

**Backend Setup:**

This generates all required boilerplate code based on the OpenAPI specification.

```bash
mvn clean install -DskipTests && mvn verify -Dopenapi.codegen.skip=false -DskipTests

```

**Frontend Setup:**

This generates the model and service layer of the web application.

```bash
cd webapp && pnpm install && pnpm openapi-generate && cd -

```

---

### Running the Application

#### Docker Services

All infrastructure and backend services can be managed via Docker. Before starting any services, navigate to the `docker` directory:

```bash
cd docker

```

If you only need to run the standard application, starting the **core** backend service is sufficient.

**Core Service**

Initializes the primary backend service:

```bash
make core

```

**Pipeline Service**

Initializes the pipeline backend service. You can choose to start it clean or pre-loaded with test data:

* **Empty** (No seeded data):
```bash
make pipeline-empty  

```


* **Seeded** (With test data):
```bash
make pipeline-seeded  

```



**Run All Services**

Initializes both Core and Pipeline services simultaneously:

* **Empty** (No seeded data):
```bash
make all-empty  

```

* **Seeded** (With test data):
```bash
make all-seeded  

```

> **Note:** Detailed information regarding these configurations can be found in the [Makefile](docker/Makefile).

#### Frontend Application

To start the web application interface, navigate to the frontend directory and start the development server:

```bash
cd webapp
pnpm run start

```

#### Backend Application
Start Core Backend Service
```Bash
mvn spring-boot:run -pl core -Dspring-boot.run.profiles=dev
```

Start Pipeline Backend Service
```Bash
mvn spring-boot:run -pl pipeline -Dspring-boot.run.profiles=dev
```

## Tests
Before submitting any code, ensure that all tests pass locally.

### Backend
Run the Backend Unit and Integration tests:
```Bash
 mvn verify
```
### Frontend
The frontend utilizes [Vitest](https://vitest.dev/) for unit testing and [Playwright](https://playwright.dev/) for End-to-End (E2E) testing. All frontend test commands should be run from within the webapp directory.
To move there use:
```bash
cd webapp
```

#### Unit Tests

Run the headless Vitest suite:
```Bash
pnpm run test
```

#### End-to-End (E2E) Tests

Run Playwright tests headlessly:
```Bash
pnpm run e2e:headless
```

Run Playwright tests with a visible browser:
```Bash
pnpm run e2e:headed
```

View the HTML report generated by Playwright after a test run:
```Bash
pnpm run e2e:report
```


## Pull Request Process

When you are ready to make a change, please follow this standard workflow:

1. **Branch off:** Create a new branch from `main` for your feature or bugfix.
2. **Commit your changes:** We use conventional commits
3. **Test locally:** Ensure your local development server runs properly and all existing linters and tests pass.
4. **Update documentation:** If your changes affect the API, dependencies, or UI, update the relevant `README.md` or specifications.
5. **Open a Pull Request:** Submit your PR against the `main` branch. Include a brief summary of your changes and link any relevant issue tickets.
6. **E2E Tests Action** You need to add the E2E-Tests label to your PR to run them in the Pipeline.
7. **Code Review:** Wait for maintainers to review your code. Be open to feedback and ready to make requested adjustments.
