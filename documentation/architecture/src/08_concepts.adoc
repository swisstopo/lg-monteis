== Querschnittliche Konzepte
=== Architekturmuster: Domain-Driven Design & CQRS-Lite (Read/Write Flows)

Die Architektur der Applikation unterscheidet strikt zwischen schreibenden Operationen (Zustandsänderungen) und reinen Lese-Operationen (Datenaggregation und Views). Dieses Muster orientiert sich am CQRS-Prinzip (Command Query Responsibility Segregation) kombiniert mit Domain-Driven Design (DDD).

Diese Trennung verhindert "Frankenstein"-Domänenmodelle, maximiert die Performance bei Lesezugriffen und schützt komplexe Geschäftsregeln bei Schreibzugriffen.

==== Der Write-Flow (Commands & Mutationen)
Gilt für alle Endpunkte, die den Systemzustand verändern (`POST`, `PUT`, `PATCH`, `DELETE`).

Hierbei wird ein striktes Domain-Driven Design erzwungen. Das Ziel ist der Schutz der Geschäftslogik ("Always-Valid Domain Model").

* *Ablauf:* Web Request -> `Controller` -> `WriteDto` -> `Mapper` -> *`Domain Entity`* -> `Service` -> `Repository` (jOOQ) -> `mapper` -> `Datenbank`.
* *Regeln:*
** DTOs werden vom Mapper in Rich-Domain-Objekte (z.B. `Sensor`, `Formula`) übersetzt.
** Validierungsregeln und Business-Logik leben ausschliesslich in der Domäne.
** Die Services und Repositories arbeiten ausschliesslich mit den Domänen-Objekten, niemals mit DTOs.
** Repositories sind für das Mapping von der Domäne in die Datenbankstruktur (via jOOQ) zuständig.

==== Der Read-Only Flow (Queries & Dashboards)
Gilt für Endpunkte, die rein lesend auf aggregierte Daten zugreifen (z.B. das `overview` Modul oder Tabellen-Ansichten über mehrere Fachbereiche hinweg).

Hierbei wird die Domänenschicht *bewusst übersprungen*. Da bei reinen Leseoperationen kein Zustand geschützt werden muss, wäre die Instanziierung von Domänenobjekten reine Ressourcenverschwendung.

* *Ablauf:* Web Request -> `Controller` -> `Service` -> `QueryRepository` (jOOQ) -> *`ReadDto`*.
* *Regeln:*
** Es werden *keine* Domänen-Objekte (`Entities`) verwendet.
** Das Repository nutzt die volle Stärke von jOOQ, um hochoptimierte SQL-Joins (z.B. über Sensoren, Experimente und Boreholes) auszuführen.
** Module, die dieses Pattern nutzen, verwenden den Suffix `QueryRepository` (statt `Repository`), um visuell zu signalisieren, dass hier kein Aggregate-Root zurückgegeben wird.

==== Zusammenfassung der Vorteile
* *Performance:* Native SQL-Joins direkt in DTOs ohne den Overhead von Objekt-Instanziierungen im Java-Speicher.
* *Saubere Domäne:* Fachobjekte wie `Sensor` bleiben fokussiert auf ihre eigenen Business-Regeln und werden nicht mit UI-spezifischen Feldern (z.B. `experimentName`) verschmutzt.

=== Core-API Datenzugriff & Sicherheit
Das Ziel dieser Architektur ist es, den Business-Code (Service Layer) zu 100 % sauber von Berechtigungsprüfungen zu halten. Die Sicherheit wird stattdessen an den Rändern des Systems durchgesetzt.

- **Der Security Context (`ScopedValue`) & Access Levels:** Ein Spring-Filter fungiert als Trichter (Funnel). Er normalisiert die eingehende Authentifizierung – unabhängig davon, ob es sich um ein menschliches JWT oder einen Maschinen-API-Key (z.B. von der Data Pipeline) handelt – in einen systemweiten Zugriffslevel (`USER`, `ADMIN`, `DATA_PIPELINE`). Dieser Level sowie die für User extrahierten Entitäts-IDs (Lese- und Schreibrechte) werden in einem modernen, threadsicheren Java `ScopedValue` (dem Nachfolger von `ThreadLocal`) für die Dauer des Requests gespeichert.

- **Die "Master Security Lineage" View:** Um komplexe Hierarchien und M:N-Beziehungen in der Datenbank abzubilden, ohne den Java-Code aufzublähen, wird eine zentrale SQL-View erstellt. Diese View mappt alle Entitäten flach auf ihre jeweiligen "Root-Rollen-IDs".

- **Der jOOQ `VisitListener` (Für Lese-/Änderungszugriffe):** Dieser jOOQ-Interceptor fängt automatisch jedes `SELECT`, `UPDATE` und `DELETE` ab, bevor es zur Datenbank geschickt wird. Er liest den Zugriffslevel und die IDs aus dem `ScopedValue` und trifft situationsbedingte Routing-Entscheidungen: `ADMIN`-Anfragen passieren ungefiltert, `DATA_PIPELINE`-Anfragen werden hart auf spezifizierte Tabellen (wie Sensordaten) isoliert, und bei regulären `USER`-Anfragen hängt er transparent eine `WHERE`-Bedingung an das SQL an, die auf Basis der Master-View filtert. Der Entwickler schreibt im Service nur `fetch()`, aber die Datenbank führt automatisch das korrekt gefilterte SQL aus.

- **Der jOOQ `RecordListener` (Für INSERTs):** Da `INSERT`-Statements keine `WHERE`-Klausel haben, greift hier ein zweiter Interceptor. Er prüft den Java-Payload direkt vor dem Speichern ab und wirft eine Exception, falls ein standard `USER` versucht, Daten für eine Entität anzulegen, für die er keine Schreibrechte besitzt.


=== Core-API Optimistic Locking und Versionierung/Auditierung

==== 1. Technischer Hintergrund: Optimistic Locking

Im Gegensatz zum Pessimistic Locking (bei dem Datensätze explizit auf Datenbankebene über `SELECT ... FOR UPDATE` gesperrt werden), geht das Optimistic Locking davon aus, dass parallele Schreibzugriffe auf denselben Datensatz selten sind. Die Konflikterkennung wird bis zum eigentlichen Schreibvorgang verzögert.

Der Mechanismus: Die Tabelle erhält eine zusätzliche Spalte (z.B. `VERSION` als Integer oder einen Zeitstempel).

Der Lesezugriff: Beim Laden eines Datensatzes wird diese Version mitgelesen.

Der Schreibzugriff: Beim Speichern wird ein `UPDATE` abgesetzt, das die alte Version in der `WHERE`-Klausel voraussetzt und die Version im `SET`-Block inkrementiert:
`UPDATE tabelle SET daten = 'neu', version = 2 WHERE id = 123 AND version = 1;`

Die Prüfung: Die Datenbank meldet zurück, wie viele Zeilen aktualisiert wurden (Rows Affected). Ist dieser Wert 0, bedeutet dies, dass ein anderer Prozess den Datensatz in der Zwischenzeit modifiziert hat. In diesem Fall wirft der ORM oder Query-Builder (hier jOOQ) eine Exception (z.B. `DataChangedException`), was in der Regel zu einem Rollback der Datenbanktransaktion führt.

==== 2. Architektur und Trennung der Zuständigkeiten

Die gewählte Schichtenarchitektur trennt das Concurrency Management (jOOQ) strikt vom Audit-Logging (JaVers).

REST-Schicht (DTOs): Dient als Ein- und Ausgangspunkt. Das DTO transportiert die Nutzdaten inklusive der vom Client zuletzt gelesenen Version.

Service-Schicht (Domain Models): Führt die reine Geschäftslogik aus. Diese Schicht arbeitet mit domänenspezifischen Java-Objekten. Ein Spring AOP-Aspekt (Aspektorientierte Programmierung) überwacht hier die Service-Methoden (z.B. getriggert durch @AfterReturning oder eine Custom Annotation).

Persistenz-Schicht (jOOQ Records): Übernimmt das Mapping vom Domain Model auf den jOOQ UpdatableRecord. Durch den expliziten Aufruf von `record.update()` generiert jOOQ das notwendige SQL für das Optimistic Locking vollautomatisch und wertet die betroffenen Zeilen aus. (Neue Datensätze werden regulär über `record.insert()` gespeichert).

Audit-Schicht (JaVers): Arbeitet vollkommen passiv. Der AOP-Aspekt greift sich das resultierende Domain Model nur dann, wenn die Methode der Business-Schicht erfolgreich und ohne Exceptions beendet wurde, und committet den Status-Diff in die JaVers-Tabellen.

==== 4. Aspekt für die Versionierung
Damit die Service-Schicht nur minimale Kenntnisse von der Versionierung haben muss, wird ein Spring AOP Aspekt verwendet. Methoden werden lediglich mit einer eigenen Annotation versehen (`@AuditChanges`), damit das betroffene Objekt vom Aspekt aufgegriffen und sauber in den Versionstabellen von JaVers abgelegt wird. Hier gilt zu beachten, dass `@AfterReturning` verwendet wird, damit beim Einfügen eines neuen Objekts dessen eindeutige ID bereits gesetzt wurde. Hier ist ein Beispiel, wie der Aspekt aussehen könnte:

[source,java]
----
import org.aspectj.lang.annotation.AfterReturning;
import org.aspectj.lang.annotation.Aspect;
import org.javers.core.Javers;
import org.springframework.security.core.context.SecurityContextHolder;
import org.springframework.stereotype.Component;

@Aspect
@Component
public class JaversAuditAspect {

  private final Javers javers;

  public JaversAuditAspect(Javers javers) {
    this.javers = javers;
  }

  @AfterReturning(pointcut = "@annotation(AuditChanges)", returning = "result")
  public void auditReturnValue(Auditable result) {

    if (result == null) {
      return;
    }

    String currentUser = "SYSTEM";
    if (SecurityContextHolder.getContext().getAuthentication() != null) {
      currentUser = SecurityContextHolder.getContext().getAuthentication().getName();
    }

    javers.commit(currentUser, result);
  }
}
----

==== Entwickler-Leitfaden: Neue Entitäten für Audit & Locking anbinden

Wenn ein neues Fachobjekt (z. B. `Experiment`) in die Architektur integriert wird, müssen folgende Schritte zwingend eingehalten werden, damit Optimistic Locking (via jOOQ) und die Historisierung (via JaVers) reibungslos ineinandergreifen:

*1. Datenbank & jOOQ (Persistenz-Schicht)*

* **Schema:** Die neue Datenbanktabelle benötigt zwingend eine Versions-Spalte (z. B. `version INTEGER NOT NULL DEFAULT 1`).
* **jOOQ Generator:** Die Konfiguration im `pom.xml` (`<recordVersionFields>VERSION|version</recordVersionFields>`) sorgt dafür, dass jOOQ das Optimistic Locking vollautomatisch anwendet.
* **Dirty Tracking (Wichtig):** Global in der Spring Boot Konfiguration muss das jOOQ-Setting `RecordDirtyTracking.MODIFIED` aktiviert sein. Dadurch prüft jOOQ via `Objects.equals()`, ob sich ein Wert *wirklich* geändert hat, bevor ein Datensatz als "dirty" markiert wird.

*2. DTO-Schicht (Data Transfer Objects als Java Records)*

* **Marker Interface:** Das DTO muss zwingend das Interface `Auditable` implementieren, damit der AOP-Aspekt es verarbeiten kann.
* **JaVers-Annotationen & Konstanten:** Das DTO _muss_ die `@Id`-Annotation auf dem Primärschlüssel tragen. Die Nutzung von `@TypeName` wird zwingend vorgeschrieben, um den Audit-Log von Java-Refactorings zu entkoppeln. Der Wert für `@TypeName` muss als `public static final String` im Record definiert werden (z. B. `JAVERS_TYPE_NAME = "Sensor"`), damit Repositories bei nativen DB-Queries typsicher darauf zugreifen können.
* **State-Transport:** Das DTO muss die Version(en) zwingend als Attribut enthalten (z. B. `version` und `formulaVersion`).

*3. Repository-Schicht*

* **Strikte Trennung:** `INSERT`- und `UPDATE`-Logik müssen in dedizierte Methoden aufgeteilt werden (Auf das generische `record.store()` wird verzichtet, da Optimistic Locking nur für Updates relevant ist).
* **Das Update-Pattern:** Jede Update-Methode muss exakt diesen Ablauf befolgen:
. Den aktuellen Datensatz (Baseline) aus der DB laden.
. *Versions-Rewind (Locking Prep):* Die Version des geladenen jOOQ-Records explizit auf die Version des eingehenden DTOs zurücksetzen (z. B. `sensorRecord.setVersion(dto.version())`).
. Neue Werte aus dem DTO blind in den Record setzen (via Setters). Dank `RecordDirtyTracking.MODIFIED` ignoriert jOOQ Setters mit identischen Werten völlig.
. `record.update()` aufrufen. jOOQ generiert nun automatisch die `WHERE id = ? AND version = ?` Klausel auf Basis der "rewound" Version. Liefert der Client eine veraltete Version, schlägt dieses Update fehl und jOOQ wirft nativ eine `DataChangedException`.
. *Ghost-Updates verhindert:* Wenn sich keine Nutzdaten geändert haben, führt `update()` dank des MODIFIED-Trackings kein SQL-Update aus und die Version bleibt unverändert. JaVers registriert dies, erkennt keine Differenz und legt automatisch keinen leeren Audit-Log an.

*4. Service-Schicht*

* **Trigger:** Die speichernde Service-Methode muss zwingend mit `@Transactional` und der eigenen Annotation `@AuditChanges` versehen werden.
* **Rückgabewert (Kritisch!):** Die Methode _muss_ das frisch gespeicherte DTO zurückgeben. Der AOP-Aspekt auditiert exakt diesen Rückgabewert, um sicherzustellen, dass die aktualisierte Version und generierte ID korrekt im Audit-Log landen.

==== 5. Umgang mit initialen Daten (Flyway Seeding & Backfill)

Da initiale Test- oder Stamm-Daten oft via Flyway-SQL-Skripte direkt in die Datenbank (`INSERT INTO ...`) geschrieben werden, umgehen sie den Spring Service Layer und somit den JaVers AOP-Aspekt vollständig. Für JaVers existieren diese Datensätze nicht, was bedeutet, dass beim ersten regulären Update über die API die Historie des Ursprungszustands verloren geht.

Um diesen "Blindspot" zu beheben, nutzt die Architektur einen **Startup Backfill Service**:

* **Event-Trigger:** Ein `@EventListener(ApplicationReadyEvent.class)` prüft beim Hochfahren der Applikation auf un-auditierte Daten.
* **jOOQ Streaming & NOT EXISTS:** Um Out-Of-Memory (OOM) Fehler und das "N+1 Query"-Problem bei grossen Datenmengen (z.B. >100.000 Sensoren) zu vermeiden, nutzt das Repository einen Datenbank-Cursor (`fetchStream()`). Über eine native `WHERE NOT EXISTS` Klausel direkt auf der JaVers-Tabelle (`jv_global_id`) wird die Filterung komplett in die Datenbank verlagert.
* **Dynamische Typisierung:** Der Abgleich in jOOQ greift dabei sicher auf die statische Konstante des DTOs zurück (z. B. `WriteSensorDto.JAVERS_TYPE_NAME`).
* **Sicherer Commit:** Gefundene unauditierte Datensätze werden vom Backfill-Service iterativ mit dem Autor `SYSTEM_SEEDER` in JaVers als `INITIAL` Snapshot committet.

=== Core-API Validierung (Validation & Error Handling)

Die Validierung ist streng in zwei Schichten unterteilt und wird über einen zentralen Mechanismus so verarbeitet, dass das Frontend (Angular) immer eine einheitliche, übersetzungsfertige Struktur erhält, während die Backend-Domäne kompromisslos geschützt bleibt.

* *Syntaktische Validierung (Formale DTO-Prüfung):*
** *Wo:* Im Web Layer (Controller-Eingang) auf den Request DTOs.
** *Wie:* Mit Jakarta-Annotationen (z.B. `@Min`, `@NotBlank`) direkt auf den Feldern oder über massgeschneiderte Cross-Field Validatoren (z.B. `@ValidFormulaState`). Als Message wird kein Text, sondern ein flacher Translation-Key (z.B. `validation.required`) hinterlegt.
** *Zweck:* Fängt formale Datenfehler ab, bevor sie gemappt werden. Resultiert in einem *422 Unprocessable Content*.
** *Hinweis zu Typen:* Falsche Datentypen (z.B. String statt Zahl) werden bereits durch den Jackson-Deserializer vor der Validierung abgefangen. Dies resultiert aus Sicherheitsgründen ("Fail Secure") in einem *400 Bad Request* mit einer generischen Fehlermeldung und einer Trace-ID, um keine internen Details preiszugeben.

* *Semantische Validierung (Business-Logik & Always-Valid Domain Model):*
** *Wo:* Direkt im Domain Model (Entities, z.B. im Konstruktor oder Setter der Klasse `Formula`).
** *Wie:* Die Domäne prüft fachliche Regeln bei der Instanziierung. Ist ein Zustand ungültig (z.B. eine Formel enthält kein 'x'), wirft die Entität einen `ValidationError`. Es ist physisch unmöglich, ein ungültiges Domänenobjekt im Backend zu erzeugen.
** *Feld- vs. Objekt-Fehler (Cross-Field Validation):*
*** *Field Errors:* Verletzen Regeln eines einzelnen Attributs. Der Fehler wird mit dem exakten Feldnamen (z.B. `expression`) geworfen und im Frontend zielgenau unter dem entsprechenden Input-Feld angezeigt.
*** *Object / Form Errors:* Verletzen Regeln, die das Zusammenspiel mehrerer Attribute betreffen (z.B. "Wenn eine ID existiert, muss auch eine Version mitgegeben werden"). In diesem Fall wird der Fehler gezielt mit einem übergeordneten Identifier wie `form` geworfen. Der Frontend-Service fängt dies ab und weist den Fehler der gesamten `FormGroup` zu, sodass er als globaler Formularfehler dargestellt werden kann.
** *Zweck:* Kapselung der Geschäftsregeln. Schlägt dies fehl, resultiert dies ebenfalls in einem *422 Unprocessable Content* mit dem betroffenen Formular-Feldnamen (oder dem übergeordneten `form` Identifier) und Message-Key.
* *Der Magische `@RestControllerAdvice` (Zentraler Exception Handler):*
** Dies ist das Herzstück der Fehlerbehandlung. Er erweitert den `ResponseEntityExceptionHandler`, um die nativen HTTP-Semantiken von Spring (wie 405 Method Not Allowed) beizubehalten, tauscht aber den Body gegen unser einheitliches `ErrorDTO` aus.
** Er definiert globale `@ApiResponse`-Annotationen, wodurch Swagger/OpenAPI automatisch die TypeScript-Interfaces für das Frontend generiert.
** *Für 422 Validierungsfehler:*
*** Er extrahiert _dynamisch_ den Namen des fehlerhaften Feldes.
*** Er liest alle Parameter aus den Annotationen (z.B. das `min` Limit aus `@Min`) komplett ohne hartcodierte Strings aus.
** *Für 400 (Client Errors) und 500 (Server Crashes):*
*** Es gilt das Prinzip der "Information Hiding". Bei Jackson-Parsingfehlern oder NullPointern wird eine UUID (Trace-ID) generiert, geloggt und an das Frontend gesendet. Der Benutzer sieht nur eine generische Meldung (z.B. "System Error"), aber Entwickler können den Fehler anhand der Trace-ID sofort in den Logs finden.

*Das Resultat für das Frontend:*

Egal ob eine DTO-Annotation fehlschlägt oder die Domäne eine Exception wirft, Angular erhält bei *422-Fehlern* immer den gleichen Payload: Den flachen Feldnamen, den Translation-Key und ein dynamisches Parameter-Objekt. Bei generischen Fehlern wird das Feld `global` verwendet.

[source,json]
----
[
  {
    "field": "age",
    "actualValue": 16,
    "messageKey": "validation.min",
    "params": {
      "value": 18
    }
  },
  {
    "field": "global",
    "actualValue": null,
    "messageKey": "error.server.internal",
    "params": {
      "traceId": "123e4567-e89b-12d3-a456-426614174000"
    }
  }
]
----

* *Übersetzungsfile im Frontend:* +
Im Angular Public Folder werden die Übersetzungsfiles platziert. Sie enthalten dieselben Schlüssel, die das Backend verwendet, und binden die dynamischen Parameter über Interpolation ein.

[source,json]
----
{
    "validation.min": "Der aktuelle Wert ist {{ actualValue }}, er muss aber mindestens {{ value }} sein.",
    "validation.required": "Dieses Feld darf nicht leer sein.",
    "error.server.internal": "Ein unerwarteter Fehler ist aufgetreten. Bitte den Support mit der Referenznummer {{ traceId }} kontaktieren."
}
----

=== 3. Skalierungsstrategien (High Throughput)

Da das externe Quell-Topic 6 Partitionen besitzt, ist die gesamte Pipeline auf maximale Parallelität über genau diese 6 Partitionen ausgelegt.

* **Kafka Connect (Ingestion):** Der MirrorSourceConnector ist mit `"tasks.max": "6"` konfiguriert. Kafka Connect baut exakt 6 parallele Threads auf, um das externe Topic ohne Bottlenecks abzusaugen.
* **Ingress Adapter (`SolExpertsRawSensorDataNormalizationService`):** Übernimmt das Entpacken der herstellerspezifischen Payloads. Nutzt `@KafkaListener(concurrency = "6")`, um die Last optimal auf die Partitionen zu verteilen.
* **Datenbank-Schicht (TimeScaleDB):** Um die Datenbank bei hohen Datenraten nicht zu überlasten, konsumiert der Processor (`SensorDataListener`) im **Batch-Modus** (`spring.kafka.listener.type=batch`, `max-poll-records=200`). Das Schreiben erfolgt über hochoptimierte jOOQ Batch-Inserts (`ctx.batchInsert()`).

=== 4. Datenintegrität & Zero-Data-Loss Architektur

Das Monteis-Pipeline-System verarbeitet Sensordaten über zwei primäre Datenströme: Den **Process-Flow** für Echtzeitdaten und den **Reprocess-Flow** für historische Datenkorrekturen. Ein zentrales Element dieser Architektur ist das Zusammenspiel aus manuellen Kafka-Acknowledgments, idempotenten Datenbank-Operationen und gezieltem Poison-Pill-Handling.

==== 4.1. Gemeinsame Basis: Orchestrator & Kafka-Offsets

Beide Flows stützen sich auf gemeinsame Komponenten und eine strikte Kafka-Konfiguration, die garantiert, dass keine Nachricht im Nirvana verschwindet.

* **Shared Transformation Orchestrator:** Um mathematische Abweichungen zu verhindern, teilen sich der Ingestion-Flow und der Backfill-Flow exakt dieselbe Logik. Dies garantiert, dass Standardisierung und Validierungs-Flags systemweit identisch berechnet werden.
* **Cache Broadcast Pattern:** Damit jeder Pipeline-Pod einen synchronen Metadaten-Cache besitzt, generiert der `SensorConfigCacheHydrator` beim Start eine zufällige UUID als Consumer Group (`cache-hydrator-#{UUID}`). Dadurch wird jedes Metadaten-Update an *jeden* laufenden Pod gebroadcastet.
* **Deaktiviertes Auto-Commit (`enable-auto-commit: false`):** Spring Boot darf Kafka-Offsets niemals im Hintergrund bestätigen. Das System verlässt sich ausschliesslich auf manuelle Bestätigungen (`ack-mode: manual`).
* **Das Guarded Acknowledgment Pattern (At-Least-Once):** In allen Listenern wird der Kafka-Offset erst explizit committet (`ack.acknowledge()`), **nachdem** die Daten sicher in die nächste Schicht oder die TimeScaleDB persistiert wurden. Fällt ein Pod aus, pausiert Kafka und liefert das Event nach dem Neustart erneut aus.

==== 4.2. Echtzeit-Verarbeitung (Process-Flow)

Dieser Flow verarbeitet neue, normalisierte Sensordaten, standardisiert sie und speichert sie in der TimeScaleDB.

* **Kafka Batching:** Der Konsum erfolgt in Batches, um den Durchsatz zu maximieren.
* **Idempotente UPSERTs:** Der `ProcessService` nutzt jOOQs `ON CONFLICT (timestamp, sensor_id) DO UPDATE`. Wenn Kafka einen Batch nach einem Absturz erneut sendet, überschreibt die Datenbank die Zeilen einfach mit exakt denselben, korrekten Daten. Es entstehen keine Duplikate.
* **Poison Pill Handling (Batch-Rettung):** Wenn ein einzelner Datensatz innerhalb eines 200er-Batches fehlerhaft ist (z. B. unlesbarer Timestamp), fängt der `ProcessService` die `TransformationException` lokal ab, loggt den fehlerhaften Rohwert und überspringt ihn (`return null`). Die verbleibenden 199 validen Datensätze werden erfolgreich in der DB gespeichert, anstatt den gesamten Batch in eine Dead Letter Queue (DLQ) zu leiten.

==== 4.3. Historische Daten-Reprozessierung (Reprocess-Flow)

Dieser Flow wird durch ein Sensor-Config-Update (`SensorConfig`) ausgelöst und muss potenziell Hunderttausende von Datensätzen aktualisieren, ohne den Speicher zu sprengen oder andere Prozesse zu blockieren. Die Logik ist in eine spezifische Orchestrator-und-Chunk-Architektur unterteilt.

* **High-Performance Guard:** Der `ReprocessService` führt zuerst eine `EXISTS`-Abfrage aus. Gibt es keine alten Versionen, wird der Prozess in Millisekunden beendet.
* **Die AOP-Proxy-Falle & Transaktionsgrenzen:** Wenn im Spring Framework eine Methode mit der `@Transactional`-Annotation aus derselben Klasse heraus aufgerufen wird, wird der Proxy umgangen. Daher ist die Logik strikt aufgeteilt: Der **`ReprocessService`** steuert die Schleife (ohne Transaktion), während der extern aufgerufene **`ReprocessChunkService`** jeden einzelnen Chunk in einer isolierten Datenbanktransaktion ausführt.
* **Iteratives Chunking (Speichersicherheit):** Eine `do-while`-Schleife ruft den Chunk-Service auf. Um JVM-OOM-Fehler zu vermeiden, erzwingt die jOOQ-Abfrage ein konfigurierbares `LIMIT` (z.B. 1000, gesteuert via `application.properties`). Die Schleife wiederholt sich, bis die Fetch-Size `0` erreicht.
* **Poison Pill Handling (Die Endlosschleifen-Falle):** Tritt bei einem historischen Datensatz ein Fehler auf (z. B. Division durch Null), darf dieser *nicht* einfach übersprungen werden. Andernfalls würde die SQL-Abfrage ihn im nächsten Schleifendurchlauf erneut laden, was zu einer Endlosschleife führt.
+
**Die Loop-Breaker Lösung:** Die Exception wird abgefangen, die `version` des fehlerhaften Datensatzes wird explizit hochgezählt und der `norm_value` wird auf `null` gesetzt. Dadurch wird die Endlosschleife gebrochen:

[source,java]
----
} catch (TransformationException ex) {
    log.error("POISON PILL REPROCESSING: Math failed for historical record of sensor {}. " +
              "Raw Value: [{}]. Bumping version to bypass infinite loop. Reason: {}",
              reading.getSensorId(), reading.getRawValue(), ex.getMessage(), ex);

    // 1. Version erzwingen zu inkrementieren, damit die DB-Abfrage den Datensatz beim nächsten Schleifendurchlauf ignoriert
    reading.setVersion(sensorConfig.getVersion().shortValue());

    // 2. norm_value auf null setzen, um sicherzustellen, dass nachgelagerte Systeme keine falsch berechneten Daten anzeigen
    reading.setNormValue(null);

    return reading;
}
----

== 5. Integration neuer externer Datenquellen

Die Architektur verwendet das **Adapter-Pattern**, um neue Datenquellen nahtlos als eigenständige "Plug-and-Play"-Komponenten zu integrieren. Um eine maximale Entkopplung und Testbarkeit zu gewährleisten, setzt die Pipeline konsequent auf **Komposition statt Vererbung** (Composition over Inheritance). Die Kern-Pipeline (alles ab dem `internal-normalized` Topic) bleibt von der Herkunft und dem ursprünglichen Format der Daten vollständig isoliert.

Wenn ein neues externes System angebunden werden soll (z. B. Sensordaten im CSV-Format auf einem FTP-Server), ist der folgende standardisierte Prozess durchzuführen:

=== 5.1. Ingress Layer konfigurieren (Kafka Connect)
Um die Architektur strikt ereignisgesteuert (event-driven) zu halten, sollte die Spring Boot Applikation externe Systeme nicht selbst aktiv pollen.

* Konfiguration eines **Kafka Connect Source Connectors** (z. B. SFTP Plugin oder SpoolDir/FilePulse).
* Der Connector pollt die externe Quelle und schreibt diese Rohdaten in ein neues, dediziertes Raw-Topic (z. B. `internal-raw-ftp-sensors`).

=== 5.2. Raw Data Model definieren
Analog zur `RawSolExpertsSensorData`-Klasse muss ein spezifisches Java-Record für die Payload der neuen Datenquelle erstellt werden (z. B. `RawFtpCsvSensorData`).

=== 5.3. Normalization Service implementieren
Die eigentliche Übersetzungslogik wird als dedizierter, isolierter Service innerhalb der Java-Applikation umgesetzt.

* **Service-Erstellung:** Erstellen eines neuen Spring `@Service` (z. B. `RawFtpCsvSensorDataNormalizationService`).
* **Komposition:** Injizieren des zentralen `NormalizedSensorDataPublisher` via Konstruktor-Injection.
* **Event-Ingestion:** Hinzufügen eines `@KafkaListener`, der auf das neu erstellte Raw-Topic lauscht.
* **Transformation:** Innerhalb der Listener-Methode die herstellerspezifischen Rohdaten parsen und in das kanonische Standardformat (`NormalizedSensorData`) umwandeln.
* **Wichtig - Guarded Acknowledgment:** Um Datenverlust (At-Least-Once Delivery) bei Fehlern oder Pod-Crashes garantiert auszuschliessen, muss das Acknowledgment-Pattern exakt wie folgt implementiert werden:
1. Für jeden extrahierten Sensorwert die Methode `publisher.publish(sensorId, canonicalPayload)` aufrufen.
2. Die resultierenden asynchronen `CompletableFuture`-Objekte in einer Liste sammeln.
3. Mittels `CompletableFuture.allOf(...).join()` blockierend darauf warten, dass der Kafka-Broker den Erhalt *aller* Nachrichten bestätigt hat.
4. **Erst nach erfolgreicher Bestätigung aller Futures** darf der Offset des ursprünglichen Raw-Topics via `ack.acknowledge()` committet werden.

=== Alternative: Zero-Code Ingestion via Kafka Connect SMTs

Für einfache, flache Datenformate (wie standardisierte CSV-Dateien via FTP) wird ein **Zero-Code-Ansatz** bevorzugt:

* **Single Message Transforms (SMTs):** Der Kafka Connect Source Connector führt einfache Transformationen (wie Spaltenumbenennungen oder Datentyp-Casts) direkt "on-the-fly" durch.
* **Direktes Routing:** Der Connector überspringt das Raw-Topic und schreibt die standardisierten Payloads direkt in das `internal-normalized` Topic.
* **Vorteile:** Keine zusätzliche Java-Codebasis, keine Unit-Tests und eine reduzierte Infrastruktur-Komplexität.

Die Implementierung eines dedizierten Java-basierten Normalisierungsservices bleibt somit ausschliesslich für Datenquellen reserviert, die komplexe Restrukturierungen, dynamisches Key-Mapping (z. B. Auflösen von verschachtelten Key-Value-Maps) oder bedingte Logik erfordern.

==== TimeScaleDB Schema (Zeitreihen & Downsampling)

Das Schema der TimeScaleDB für die hochfrequenten Zeitreihendaten umfasst neben der Hypertable auch die Continuous Aggregates (Downsampling-Buckets) und die Retention-Policies für das Daten-Lebenszyklus-Management.

[source,sql]
----
-- 1. Enum für Range-Status
-- Bleibt speichereffizient (4 Bytes) und hochgradig komprimierbar
CREATE TYPE range_category AS ENUM ('too_low', 'correct', 'too_high');

-- 2. Base Table erstellen (Raw Data)
-- Hinweis: sensor_id ist sensor_code in der Metadaten DB
CREATE TABLE sensor_reading (
    timestamp   TIMESTAMPTZ NOT NULL,
    sensor_id   TEXT NOT NULL,
    raw_value   DOUBLE PRECISION NOT NULL,
    norm_value  DOUBLE PRECISION,
    version     SMALLINT DEFAULT 1,
    status      range_category,

    -- Composite Primary Key: Erzwingt Eindeutigkeit pro Sensor & Zeitstempel
    PRIMARY KEY (timestamp, sensor_id)
);

-- 3. In eine Hypertable konvertieren (Partitionierung nach 'timestamp')
SELECT create_hypertable('sensor_reading', 'timestamp');

-- 4. Native Kompression für Originaldaten aktivieren (nach 120 Tagen)
ALTER TABLE sensor_reading SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'sensor_id'
);
SELECT add_compression_policy('sensor_reading', INTERVAL '120 days');


-- ==========================================
-- CONTINUOUS AGGREGATES (Downsampling Buckets)
-- ==========================================

-- Bucket 1: 1-Minuten-Aggregation
CREATE MATERIALIZED VIEW sensor_reading_1m
WITH (timescaledb.continuous) AS
SELECT time_bucket('1 minute', timestamp) AS bucket,
       sensor_id,
       AVG(norm_value) AS avg_value,
       MAX(norm_value) AS max_value,
       MIN(norm_value) AS min_value
FROM sensor_reading
GROUP BY bucket, sensor_id;

-- Bucket 2: 15-Minuten-Aggregation
CREATE MATERIALIZED VIEW sensor_reading_15m
WITH (timescaledb.continuous) AS
SELECT time_bucket('15 minutes', timestamp) AS bucket,
       sensor_id,
       AVG(norm_value) AS avg_value
FROM sensor_reading
GROUP BY bucket, sensor_id;

-- Bucket 3: 1-Stunden-Aggregation
CREATE MATERIALIZED VIEW sensor_reading_1h
WITH (timescaledb.continuous) AS
SELECT time_bucket('1 hour', timestamp) AS bucket,
       sensor_id,
       AVG(norm_value) AS avg_value
FROM sensor_reading
GROUP BY bucket, sensor_id;

-- Bucket 4: 1-Tages-Aggregation (Langzeitarchiv)
CREATE MATERIALIZED VIEW sensor_reading_1d
WITH (timescaledb.continuous) AS
SELECT time_bucket('1 day', timestamp) AS bucket,
       sensor_id,
       AVG(norm_value) AS avg_value
FROM sensor_reading
GROUP BY bucket, sensor_id;


-- ==========================================
-- RETENTION POLICIES (Rolling Windows)
-- ==========================================

-- Alte Aggregationen verwerfen, sobald sie aus ihrem Zeitfenster fallen:
SELECT add_retention_policy('sensor_reading_1m', INTERVAL '30 days');
SELECT add_retention_policy('sensor_reading_15m', INTERVAL '60 days');
SELECT add_retention_policy('sensor_reading_1h', INTERVAL '2 years');
-- Anmerkung: Die 'sensor_reading_1d' View behält Daten dauerhaft.

-- ==========================================
-- REFRESH POLICIES (Background Jobs für Continuous Aggregates)
-- ==========================================

-- 1-Minuten-Bucket: Aktualisiert sich jede Minute im Hintergrund
SELECT add_continuous_aggregate_policy('sensor_reading_1m',
    start_offset => NULL,
    end_offset   => INTERVAL '1 minute',
    schedule_interval => INTERVAL '1 minute');

-- 15-Minuten-Bucket: Aktualisiert sich alle 15 Minuten
SELECT add_continuous_aggregate_policy('sensor_reading_15m',
    start_offset => NULL,
    end_offset   => INTERVAL '15 minutes',
    schedule_interval => INTERVAL '15 minutes');

-- 1-Stunden-Bucket: Aktualisiert sich stündlich
SELECT add_continuous_aggregate_policy('sensor_reading_1h',
    start_offset => NULL,
    end_offset   => INTERVAL '1 hour',
    schedule_interval => INTERVAL '1 hour');

-- 1-Tages-Bucket: Aktualisiert sich einmal pro Tag (z.B. Nachts)
SELECT add_continuous_aggregate_policy('sensor_reading_1d',
    start_offset => NULL,
    end_offset   => INTERVAL '1 day',
    schedule_interval => INTERVAL '1 day');
----

=== Teststrategie

Um die Stabilität und Datenintegrität von MONTEIS zu gewährleisten, verfolgen wir eine schichtenbasierte Teststrategie. Der Fokus liegt darauf, die gesamte Funktionalität und das fachliche Verhalten der Applikation rigoros abzusichern, ohne durch fragile Layout-Tests unnötigen Wartungsaufwand zu erzeugen.

Um eine konstant hohe Code-Qualität zu garantieren, wird systemübergreifend **SonarQube** eingesetzt. Eine Mindestabdeckung dient als Guardrail, wobei die Aussagekraft der Tests (fachliche Relevanz, Fehlererkennungsfähigkeit) höher gewichtet wird als die reine Prozentzahl.

==== Frontend Testing
Im Angular-Frontend liegt der Fokus auf der Überprüfung der isolierten UI-Logik (z. B. State-Management, Filter-Logik, Datenaufbereitung für Charts).

* **Technologie:** Wir verwenden das moderne, von Angular unterstützte **Vitest** für die Unit-Tests.
* **Fokus:** Es wird ausschliesslich Logik getestet. Reine Darstellungsaspekte (Layout, CSS, Pixel-Perfektion) werden bewusst von Unit-Tests ausgeschlossen.

==== Backend Testing (Spring Boot)
Das Backend wird strikt in verschiedene Test-Ebenen unterteilt (Test-Pyramide), um schnelle Feedback-Zyklen und realitätsnahe Datenbanktests optimal zu kombinieren. Als Basis dient hierbei die native **Spring Boot Test Suite**.

* **Web-Schicht (Controller Tests):** Für die REST-Schnittstellen verwenden wir das Spring Boot Slice-Testing (`@WebMvcTest`) in Kombination mit `MockMvc`. Dadurch testen wir das korrekte HTTP-Routing, Input-Validierungen, HTTP-Statuscodes und die JSON-Serialisierung/-Deserialisierung. Die zugrundeliegende Business-Logik (Services) wird auf dieser Ebene via `@MockBean` gemockt, um den Application Context klein und performant zu halten.
* **Business-Logik (Unit Tests):** Die eigentliche Fachlogik (Services, Mapper und Configurations) wird isoliert über klassische Unit-Tests mit **JUnit 5** und **Mockito** getestet. Externe Systemgrenzen und die Datenbank (Persistenzschicht) werden auf dieser Ebene konsequent gemockt.
* **Persistenzschicht (Integration Tests):** Die Persistenzschicht (jOOQ-Repositories) wird von den normalen Unit-Tests ausgeschlossen. Stattdessen nutzen wir **Spring Boot Testcontainers**, um die echten Datenbanken als isolierte Docker-Container hochzufahren.
** **Architektur-Fokus (FDW):** Um den reibungslosen Datenabruf zu testen, fahren die Testcontainers sowohl eine PostgreSQL- als auch eine TimeScaleDB-Instanz hoch. Im Rahmen des Test-Setups wird der FDW-Tunnel zwischen den Containern initialisiert.
** **Testdaten-Management:** Das Schema und die initialen Testdaten für die Container werden pro Test-Suite automatisiert über spezifische **Flyway-Skripte** aufgebaut.

==== End-to-End Testing (E2E)
Um die isolierten Komponenten-Tests zu komplementieren und das Zusammenspiel des gesamten Systems (Frontend + Backend + Datenbank) zu validieren, setzen wir ebenenübergreifende E2E-Tests ein.

* **Technologie:** **Playwright**
* **Infrastruktur & Setup:** Um eine verlässliche und reproduzierbare Testumgebung für die E2E-Tests zu schaffen, implementieren wir ein dediziertes **Spring Boot Testing-Profil**. Startet man das Backend mit diesem Profil, fährt es automatisch die benötigten Testcontainers (PostgreSQL & TimeScaleDB) hoch und befüllt sie via Flyway. Das Frontend kann anschliessend gegen dieses vollständig gekapselte Backend getestet werden.
* **Fokus:** Getestet wird das Verhalten der Applikation anhand der wichtigsten User-Workflows (z. B. Sensor-Filterung, Sensor erstellen, Experiment erstellen, etc.). Auch hier steht das funktionale Verhalten im Vordergrund, nicht das Layout.

=== Entwickler-Workflow & CI/CD Pipeline

Um eine konsistente Code-Qualität zu garantieren, den Review-Prozess zu verschlanken und die Server-Ressourcen effizient zu nutzen, wird der Code-Lebenszyklus von lokalen Git-Hooks bis hin zu automatisierten GitHub Actions strikt orchestriert.

==== Lokaler Workflow (Git Hooks)
Bevor Code in das Repository gepusht wird, greifen lokale Git-Hooks , um formelle Standards bereits auf dem Entwickler-Rechner zu erzwingen:

* **Formatter Pre-Commit Hook:** Dieser Hook formatiert geänderte Dateien automatisch und fügt sie dem Staging-Bereich (`git add`) wieder hinzu. Dadurch wird sichergestellt, dass der Commit exakt den sauber formatierten Stand enthält.
** **Frontend (`.ts` / `.html`):** Formatierung und Linting via **Prettier** und **ESLint**.
** **Backend (`.java`):** Formatierung via **Spotless** unter strikter Verwendung des **Google Java Formats**.
* **Commit-Message Hook (`commit-msg`):** Prüft die Commit-Nachricht vor dem Speichern auf das Einhalten der **Conventional Commits** (TBD).

==== Continuous Integration (Pull Request Workflows)
Um die Ressourcen (und Kosten) der Pipeline zu schonen, werden ressourcenintensive CI-Jobs **nicht bei jedem einzelnen Push**, sondern ausschliesslich bei der Erstellung oder Aktualisierung von **Pull Requests (PRs)** ausgeführt.

* **Format Checker:** Prüft den gesamten Codebestand, um sicherzustellen, dass keine unformatierten Dateien die lokalen Hooks umgangen haben (z. B. durch `git commit --no-verify`).
* **Full Test Suite:** Führt die oben definierte Teststrategie (Unit, Controller, Integration via Testcontainers, E2E) vollständig aus.
* **Security Scanning (Trivy):** Eine dedizierte GitHub Action nutzt **Trivy** für automatisierte Sicherheitsüberprüfungen. Dabei werden die Projekt-Abhängigkeiten, Dockerfiles und Container-Images auf bekannte Schwachstellen (CVEs) und Fehlkonfigurationen (IaC) gescannt. Gefundene kritische Sicherheitslücken (z. B. Severity "HIGH" oder "CRITICAL") blockieren den Pull Request.
* **SonarQube Quality Gate:** Eine dedizierte GitHub Action (`sonar-scan`) analysiert den Code auf Bugs, Vulnerabilities und Code Smells. Anschliessend prüft die Action den Status des SonarQube Quality Gates. Ein Pull Request kann nur bei grünem Quality Gate gemerged werden.

==== Continuous Deployment & Release-Workflows

Sobald Code-Änderungen in den `main-branch` integriert werden oder offizielle Versionen anstehen, übernehmen automatisierte GitHub Actions das Deployment. Das Setup folgt strikt dem Build-Once-Prinzip für Produktions-Releases und einem GitOps-Ansatz für die Infrastruktur.

Es sind drei dedizierte Workflows definiert:

* **Workflow 1: Auto-Deploy auf DEV (Bleeding Edge)**
** **Trigger:** Automatisch bei jedem Push oder Merge in den `main`-Branch.
** **Ablauf:** Der Workflow baut sofort und kompromisslos ein neues Docker-Image aus dem aktuellen Stand des Codes und deployt dieses auf die `DEV`-Umgebung. Diese Stage spiegelt den aktuellsten Entwicklungsstand wider und muss nicht zwingend stabil sein.

* **Workflow 2: Create Release (Deployment auf STAGING)**
** **Trigger:** Manuell (GitHub Workflow Dispatch).
** **Ablauf:**
1. Der ausführende Nutzer wählt die Art des Releases (Major, Minor oder Patch basierend auf dem letzten existierenden Git-Tag).
2. Der Workflow aktualisiert die Version im Code (z. B. via `mvn clean install -Drelease=vX.X.X` im Backend).
3. Er erstellt den neuen Git-Tag sowie ein offizielles GitHub Release.
4. Die Anwendung wird kompiliert und das neue Docker-Image wird gebaut.
5. Das frisch gebaute Image wird gestartet und durchläuft sofort die komplette End-to-End (E2E) Test-Suite.
6. Nur wenn die E2E-Tests erfolgreich sind, wird das Image in die Container Registry (Amazon ECR) gepusht und automatisch auf der `STAGING`-Umgebung deployt.

* **Workflow 3: Promote to PROD (GitOps)**
** **Trigger:** Manuell (GitHub Workflow Dispatch).
** **Ablauf:** Dieses Skript baut bewusst *keine* neuen Images, um ungetestete Seiteneffekte zu vermeiden.
1. Der Nutzer wählt einen bereits existierenden Release-Tag aus dem Dropdown.
2. Der Workflow sucht in der Amazon ECR nach dem exakt passenden, auf Staging bereits validierten Docker-Image.
3. Der Workflow checkt das separate Produktions-Infrastruktur-Repository aus.
4. Er aktualisiert dort lediglich den Image-Tag in der `kustomize.yaml` und committet die Änderung.
5. Durch diesen Commit im Infra-Repo wird das eigentliche Kubernetes-Deployment auf `PROD` (GitOps-Ansatz) angestossen.
