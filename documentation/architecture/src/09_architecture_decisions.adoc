| Kriterium | Beschreibung

| Titel
| ADR: Einsatz von Anchore Syft und Grant für Open-Source-Lizenzkonformität

| Datum
| 08.06.2026

| Status
| entschieden

| Kontext & Problem
| Vermeidung von riskanten Open-Source-Lizenzen (z.B. Copyleft wie GPL-3.0) in Node.js- und Java-Projekten. Benötigt wird eine automatisierte, tiefgehende Prüfung innerhalb der CI/CD-Pipeline vor dem Merge.

| Entscheidung
| Einsatz von Anchore Syft zur Generierung der Software Bill of Materials (SBOM) im SPDX-JSON-Format und Anchore Grant zur lokalen Prüfung gegen eine .grant.yml-Richtlinie.

| Begründung
| Performance durch Entkopplung: Syft führt tiefe Netzwerkscans durch (Ergebnis wird gecached), Grant prüft das lokale SBOM in Millisekunden.

Tiefe Auflösung: Native Unterstützung für die Auflösung von Maven-Parent-POMs und externen npm-Registries.

SaaS-frei & Sicher: Die Prüfung läuft komplett lokal im GitHub Actions Runner ab. Keine Übertragung von Code oder Daten an Drittanbieter.

Configuration as Code: Richtlinien sind versioniert und auditierbar direkt im Repository gespeichert.

| Konsequenzen
| Schnelle CI-Pipelines und sofortiges Blockieren bei Lizenzverstößen. Erfordert als Trade-off die manuelle Pflege einer ignore-packages-Liste für fehlerhafte Lizenz-Strings oder interne CI-Tools.
|===

| Kriterium | Beschreibung

| Titel
| ADR: Durchsetzung der Leseberechtigungen (Row-Level Security) auf Datenbankebene statt in der Applikationsschicht

| Datum
| 17.07.2026

| Status
| entschieden

| Kontext & Problem
| Ein früheres Konzept sah einen jOOQ-seitigen `VisitListener`/`RecordListener` vor, der Queries abfängt und Berechtigungen im Code prüft. Nachteile: jeder neue Query-Pfad müsste ihn korrekt durchlaufen, die per Foreign Data Wrapper eingebundene TimescaleDB-Tabelle unterstützt ohnehin keine native RLS, und Berechtigungslogik im Code ist schwerer zu auditieren als deklarative DB-Policies.

| Entscheidung
| Leseberechtigungen werden über native PostgreSQL Row-Level-Security-Policies auf `sensors`/`experiments` sowie eine security-barrier-View (`sensor_reading_secured`) für die FDW-Tabelle durchgesetzt. Ein custom jOOQ `ConnectionProvider` (`RlsConnectionProvider`) schreibt dafür bei jeder Connection-Beschaffung die aktuelle `Authentication` transaktionslokal auf die Verbindung; die Policies lesen diesen Kontext serverseitig aus. Die `Authentication` selbst wird von einem custom `Converter` (`JwtSecurityAuthenticationConverter`) aus dem JWT gebaut — Rollen werden zu `api:*`-Authorities (via `RealmRoleAuthoritiesConverter`), Sub und Experiment-IDs landen in einem eigenen `SecurityPrincipal` als Authentication-Principal — und liegt für die Dauer des Requests im Standard-`SecurityContextHolder` von Spring Security, ohne eigenen Propagierungsmechanismus.

| Begründung
| Enforcement in der Datenbank ist unabhängig vom Zugriffspfad (jOOQ, native SQL, künftige Tools) und lässt sich nicht durch einen vergessenen Code-Check umgehen. Der transaktionslokale Kontext verhindert ein Durchsickern zwischen Anfragen bei Connection-Pooling.

| Konsequenzen
| RLS-relevante Repository-Methoden müssen innerhalb einer Spring-Transaktion laufen (`@Transactional(readOnly = true)`), sonst wird der Kontext nie gesetzt bzw. gelesen. Schreiboperationen bleiben auf DB-Ebene permissiv, da alle Zugriffe dieselbe Rolle (`core_app`) nutzen — siehe folgende ADR zur Trennung von Read- und Write-Autorisierung.
|===

| Kriterium | Beschreibung

| Titel
| ADR: Trennung von Lese- und Schreibautorisierung — RLS für Reads, RBAC für Writes

| Datum
| 17.07.2026

| Status
| pending

| Kontext & Problem
| Alle DB-Zugriffe laufen über dieselbe Rolle (`core_app`); DB-Grants können daher nicht zwischen einzelnen Nutzern unterscheiden — sie eignen sich für die RLS-Lesefilterung, nicht aber für nutzerspezifische Schreibrechte.

| Entscheidung
| Lesen bleibt vollständig RLS-gefiltert (siehe vorherige ADR). Schreiben soll stattdessen klassisch rollenbasiert (RBAC) auf Applikationsebene erfolgen, z. B. via `@PreAuthorize("hasRole('ADMIN')")` auf den Controller-Methoden, basierend auf denselben Rollen, die auch der `SecurityContext` nutzt.

| Begründung
| Schreiben braucht meist nur ein einfaches rollenbasiertes Ja/Nein, während Lesen feingranular pro Experiment gefiltert werden muss. Das trennt die Verantwortlichkeiten klar: Datenbank filtert Daten, Applikation autorisiert Aktionen.

| Konsequenzen
| Mit dem Kunden ist noch offen, wie granular Schreibrechte pro Rolle/Experiment sein müssen — die konkrete `@PreAuthorize`-Ausprägung pro Endpunkt ist deshalb noch nicht final.
|===
